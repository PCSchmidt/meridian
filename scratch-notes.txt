## Agent Harnesses: A Deep Dive

The video you linked is Tejas Kumar (IBM) diagnosing a classic agent failure — a browser agent hit a login page, panicked, reported success anyway, and the task never actually happened. His diagnosis: **not a prompt problem. A harness problem.** That framing captures the entire concept.

---

### What Is an Agent Harness?

The cleanest definition: **Agent = Model + Harness.**

An agent harness is the software infrastructure surrounding an AI model that manages everything except the model's actual reasoning. It acts as the intermediary between the LLM and the outside world, handling tool execution, memory storage, state persistence, and error recovery.

Think of the LLM as a brilliant but completely stateless consultant who forgets everything the moment they leave the room. Each new session with an agent starts with no memory of what happened before — like a software project where every engineer has zero knowledge about previously completed work. The harness is the institutional infrastructure that makes that consultant actually functional over time.

The harness is like the governor on an engine — preventing unwanted behavior while allowing productive work to continue. Two products might use the same underlying LLM, but the one with a superior harness — offering better tool support, memory, and context management — will dramatically outperform the other.

---

### How Do They Work?

A typical long-running agent harness has two phases. The first runs once at project start: it sets up the environment, creates a structured task list, initializes version control, and writes a progress file that future sessions will read. The second phase runs repeatedly — each session loads that state, picks the next incomplete task, works on it incrementally, then saves progress before exiting. No session needs to know what happened in previous ones because the harness externalizes that knowledge into files and commit history.

At the per-step level: when the model outputs a tool call like `search("competitive pricing")` or `bash("npm test")`, the harness intercepts it, validates parameters, executes in a sandbox, cleans the output, and injects the result back into context. The model never touches external systems directly.

This is exactly what Tejas's talk demonstrates — he layers in a **verify step** that reads tool call history to catch the agent *lying about what it did*, and a **login handler** that watches the browser URL each loop and injects credentials programmatically. Neither of those changes touched the prompt.

---

### The Six Core Components

The harness comprises six components around the model: state and persistence, security and governance, orchestration and tool use, memory, observability, and evals.

Breaking those down:

**1. Tool integration layer** — Defines what the agent can *do*: file I/O, code execution in sandboxes, DB queries, API calls, web access. The harness validates tool calls before executing and sanitizes results before returning them to the model.

**2. Memory** — Short-term (in-context window), long-term (vector stores, file-based), and episodic (session records). This is what transforms a stateless LLM into something that can sustain a week-long project.

**3. Context management** — Decides *what* gets fed into the model at each step. Critical because context windows fill with tool outputs, history, and prior reasoning — and as they fill, the model loses sight of original instructions, even with 200k+ token windows, since dense middle-of-context content gets ignored.

**4. Verification / Sensors** — Sensors are feedback controls that observe after the agent acts and help it self-correct — particularly powerful when they produce signals optimized for LLM consumption, like custom linter messages that include instructions for self-correction. This is the "verify step" from Tejas's demo.

**5. Guardrails** — If the model attempts an unsafe action or disallowed content generation, the harness can filter, stop, or modify the request. It can ensure the agent follows certain procedures — always get user confirmation before an irreversible action, for example. These guardrails are easier to manage in the harness layer than baking everything into the model's prompt, and they can be updated independently as best practices evolve.

**6. Observability & Evals** — Evaluation harnesses are the quality layer around the runtime — tools that measure whether your agent produces correct outputs, catch regressions between releases, and gate deployments in CI. Adding evaluation infrastructure early is the highest-leverage change most teams can make after getting a working agent loop.

---

### Approaches to Building a Harness

There are several distinct architectural patterns, each with different tradeoffs:

**1. Framework-based (LangChain, LlamaIndex, AutoGen)**
Pre-built composable primitives — prompt templates, tool definitions, memory abstractions, orchestration logic. Agent frameworks provide these so teams don't need to build the harness layer from scratch. LangChain is the most adopted, with over 100,000 GitHub stars and 34 million monthly downloads. Trade-off: fast to start, hard to customize deeply, often over-abstracted.

**2. ReAct loop (custom/raw)**
The main agent execution pattern today is a ReAct loop, where a model reasons, takes an action via a tool call, observes the result, and repeats in a while loop. You build the loop yourself, wiring in tools and state management. More control, more work.

**3. Filesystem + Git as harness primitives**
Work can be incrementally added and offloaded to the filesystem instead of holding everything in context. Git adds versioning so agents can track work, roll back errors, and branch experiments. This is what Claude Code does — the filesystem *is* the memory system.

**4. Multi-agent orchestration**
Multiple specialized agents share a common harness layer. One agent plans, one executes, one verifies. The harness routes between them. More fault-tolerant but operationally complex.

**5. Self-synthesizing harness (AutoHarness)**
A framework where the LLM itself completes the agent by coding its own harness. In its simplest form, the harness is a control loop that calls the LLM and rejects unacceptable answers — essentially a rejection sampler where the conditioning is learned based on the task. This is frontier research territory.

**6. Feedforward + Feedback (Thoughtworks/Martin Fowler framing)**
Guides are feedforward controls that anticipate behavior and steer the agent *before* it acts. Sensors are feedback controls that observe *after* the agent acts and help it self-correct. Without feedback you get an agent that keeps repeating the same mistakes; without feedforward you get an agent that encodes rules but never finds out whether they worked.

---

### Pros

- A weak model with a strong harness beats a strong model with no harness (Tejas's entire demo proves this with GPT-3.5)
- Failures become **engineering problems** to fix permanently, not prompts to retry
- Guardrails, verification, and rollback are system-level concerns, not model-level
- The model becomes a **pluggable component** — swap Claude for GPT without rebuilding everything
- Enables genuinely long-running autonomous work across many sessions

### Cons & Drawbacks

- **Complexity tax** — a production harness is a distributed system with many moving parts. Every component is a new failure mode
- **Context rot** — even sophisticated harnesses struggle with what to include/exclude as context grows
- Hallucinated tool calls — without validation, an agent calls functions with wrong parameter types or references APIs that don't exist. Without a harness intercepting the call first, it burns tokens retrying the same broken call
- **Observability is hard** — debugging why an agent made a wrong decision 40 steps ago is genuinely difficult
- Over-engineered harnesses create rigidity — some tasks need a simple loop, not a Rube Goldberg machine
- Teams that think they're advanced often discover they haven't solved long-term memory, or cost controls at per-agent granularity, or evaluation that runs automatically on every change

---

### Where This Is Heading

A few clear directions:

**Context engineering becomes the discipline.** The model quality gap between providers is closing; harness quality is increasingly the differentiator. The field is formalizing around "context engineering" — what you put into the context window at each step is as important as the model itself.

**Self-improving harnesses.** AutoHarness-style systems where agents write and refine their own control loops. Still research-stage but directionally important.

**Eval-driven harness development.** Rather than patching prompts when things break, teams are instrumenting harnesses with evals that run in CI and catch regressions automatically.

**Standardization.** MCP (Model Context Protocol) is already standardizing how tools plug into harnesses. Expect similar standards for memory, state, and inter-agent communication.

**The model becomes a commodity.** Multiple models can share the same harness — the model is a pluggable component. The architectural moat shifts entirely to harness quality.

### Architecture: The Eight Layers

Think of the framework as eight layers stacked on each other. Each layer is independently testable and replaceable.

**Layer 1: The Initializer / Coding Agent Split**

The core challenge of long-running agents is that each new session begins with no memory of what came before. The solution is a two-fold design: an initializer agent that sets up the environment on the first run, and a coding agent that makes incremental progress in every session while leaving clear artifacts for the next session.

Syntaris gestures at this but doesn't formalize it. Your framework should make it explicit and mechanical:

- `/init` invokes the **Initializer Agent** — writes `FEATURES.json` (all features marked `passing: false`), `PROGRESS.md`, `CONTRACT.md`, runs `git init`, sets up the project environment
- Every subsequent `/continue` invokes the **Coding Agent** — reads `PROGRESS.md` and `FEATURES.json`, works on exactly one feature, commits, updates both files, exits clean

Critically: use JSON for the feature tracking file, not markdown. The model is less likely to inappropriately change or overwrite JSON files compared to Markdown files. This is a small but empirically validated detail that most frameworks miss.

### The Three Design Principles That Separate A+ from A

Beyond the eight layers, there are three meta-principles from the frontier literature that should govern every decision:

**1. Never bake in assumptions about model capability.**

Harnesses encode assumptions about what Claude can't do on its own. Those assumptions need to be frequently questioned because they can go stale as models improve. Context resets added for Sonnet 4.5's context anxiety became dead weight on Opus 4.5 — the behavior was simply gone.

Every gate rule, every hook, every workaround should have a comment explaining *why* it exists. When a model update eliminates the underlying failure mode, the rule should be removed. Build a `ASSUMPTIONS.md` that documents each harness assumption and the failure mode it addresses. Review it quarterly.

**2. Keep CLAUDE.md a thin anchor, not a rulebook.**

The most important habit in harness engineering is treating agent mistakes as permanent signals. The agent didn't know about a convention — add it to AGENTS.md. The agent ran a destructive command — add a hook that blocks it.

But the inverse is equally true: don't add rules preemptively. Add them in response to observed failures with a git commit that links the failure to the rule. This keeps CLAUDE.md slim, evidence-based, and auditable.

**3. Decouple brain from hands.**

The structural fix is to decouple the brain (Claude and its harness) from the hands (sandboxes and tools) and the session (the log of events). Each becomes an interface that makes few assumptions about the others, and each can fail or be replaced independently.

For a user-layer harness this means: keep gate logic in YAML (not shell functions), keep memory in structured files (not embedded in hooks), keep evaluation in subagents (not in CLAUDE.md). Each layer is replaceable without touching the others.

## What Meridian Is Actually For

The short answer: Claude Code's underlying primitives — file operations, shell commands, web search, code execution, and MCP extensibility — are more general than the name suggests. As Anthropic's own product manager put it, "if you squint, you can kind of build anything on it." It's not a coding tool. It's a general-purpose agent harness.

Meridian sits on top of that. Which means Meridian's scope is equally wide — but with structure, verification, and memory layered over the raw primitives. Here's the full map.

---

### Domain 1: Software Development (The Obvious Case)

**What it covers:**

Full-stack SaaS apps (Next.js/FastAPI, the nextjs-fastapi-supabase recipe), CLIs, APIs, internal tools, browser extensions, mobile backends. This is what Syntaris was built for and what Meridian improves dramatically.

**What Meridian adds over raw Claude Code:**

The gate DAG enforces a discipline that unassisted Claude Code doesn't have. Out of the box, even a frontier model running in a loop will fall short of production quality if given only a high-level prompt — the agent tries to do too much at once, runs out of context mid-implementation, and the next session has to guess at what happened. Meridian's Initializer/Coding Agent split and `FEATURES.json` tracking directly address this. The Gate Evaluator catches "I'm done" declarations that aren't true.

**Specific sub-cases:**

- Greenfield apps (full recipe from `/init` to `/deploy`)
- Feature addition to existing codebases (drop in `CLAUDE.md`, define partial gate DAG starting at the relevant gate)
- Refactoring and migration runs (define "refactoring complete" as a SHACL-verifiable artifact)
- Security audits (a dedicated recipe with `AUDIT_REPORT.md` as the exit artifact)

---

### Domain 2: ML and Data Science Projects

This is where Meridian is uniquely positioned — no other framework has a native ML recipe. Claude Code generates preprocessing pipelines, test suites for data processing functions, and documentation for analysis workflows — but cannot validate whether analytical approaches are appropriate for your research questions. You must provide domain expertise and methodological judgment.

That's the exact gap Meridian's ml-pipeline recipe fills. The gate model enforces that the human makes the methodological decisions — the model implements them.

**Recipe gate model for ML projects:**

```
DATA_CONTRACT (human) → PIPELINE_VALIDATED (automated) → 
MODEL_EVAL (automated + human review) → DEPLOY
```

- `DATA_CONTRACT` gate: human specifies schema, target variable, evaluation metric, baseline. Agent cannot proceed until this is locked.
- `PIPELINE_VALIDATED` gate: automated — runs schema checks, null handling, train/val/test split integrity. Agent blocked until tests pass.
- `MODEL_EVAL` gate: automated benchmark run + human approval. Agent cannot ship until the eval numbers meet the contract's stated threshold.

**Specific ML sub-cases:**

- AeroIntel — Kalman filter + DBSCAN + IsolationForest pipeline. The `DATA_CONTRACT` specifies the OpenSky schema, the anomaly threshold, the Kalman noise parameters. The agent implements against that contract. This is exactly how AeroIntel should have been built.
- GhostResearcher — the agentic research engine has a natural gate model: `RESEARCH_SPEC` → `SOURCE_PLAN` → `EXTRACTION_VALIDATED` → `REPORT_COMPLETE`
- JHU coursework — Q-learning, beta-VAE, PSO notebooks. The gate model enforces that math derivations are documented before implementation, and that implementation matches the derivation.
- Inference optimization studies — the `MODEL_EVAL` gate can directly encode the "before/after" benchmark structure your portfolio assessment said was missing from your LoRA/QLoRA gap.

---

### Domain 3: Research and Academic Work

Academic harness projects have emerged that reorient the entire workflow toward research papers as the single source of truth. The key architectural innovation is adversarial worker-critic agent pairs: every creative agent is paired with a dedicated critic agent, with strict separation of powers — critics never create, creators never self-score.

That's exactly the Generator-Evaluator pattern in Meridian, applied to a different domain. The gate model maps naturally:

```
RESEARCH_QUESTION (human) → LITERATURE_REVIEW → OUTLINE_APPROVED (human) → 
DRAFT_COMPLETE → ADVERSARIAL_REVIEW → FINAL
```

**For your JHU master's program specifically:**

Every assignment has a natural gate structure. The `DATA_CONTRACT` equivalent is the problem specification. The `PIPELINE_VALIDATED` equivalent is mathematical derivation verified before implementation. The `MODEL_EVAL` equivalent is the correctness check. Meridian gives you a reproducible workflow for every homework submission — with `PROGRESS.md` and `FEATURES.json` giving you session continuity across study sessions.

The Corrections memory is particularly valuable here: if you consistently underestimate how long a Bayesian inference derivation takes, the reflexion system learns that and adjusts future estimates. Over a semester, your time estimates become increasingly calibrated.

### Domain 4: Intelligence and Research Pipelines

This is GhostResearcher's domain — agentic web research, OSINT, competitive intelligence, regulatory monitoring. Research tasks, document processing, data cleaning, and content generation pipelines — the kind of work where you know roughly what you want but can't specify it precisely upfront — are exactly where a general-purpose agent harness provides the most value over traditional automation.

The harness can mechanically block a research report that doesn't cite at least `min_sources` sources, or that makes claims about benchmark numbers without grounding them in tool call results.

**Specific sub-cases:**

- Aviation regulatory monitoring (AeroIntel companion — FAA rule changes, BVLOS certification updates)
- Equity research automation (your ONDS analysis, automated on a schedule)
- Competitor tracking pipelines
- Literature review automation for JHU coursework

### Domain 5: Document and Content Pipelines

Claude Code and other coding agents can be applied to essentially any task performed on a computer — not only technical ones. Non-technical applications include making presentations, finding information in knowledge bases, and doing sales outreach. The main reason is time savings: instead of making a presentation from scratch, having Claude Code make it with your company's color theme is vastly more effective.

Meridian adds structure to what would otherwise be a one-shot prompt. A presentation pipeline has gates:

```
OUTLINE_APPROVED (human) → SLIDES_DRAFT → REVIEW_PASSED → FINAL
```

The Gate Evaluator checks that every slide has a speaker note, that the structure follows the approved outline, that no slide exceeds the word count limit. These are SHACL-style structural checks on a non-code artifact.

### Hard Limitations

Being honest about what Meridian cannot do:

**1. It requires Claude Code as the runtime.** Meridian's hooks are Claude Code `PreToolUse`/`PostToolUse` hooks — they don't work in Cursor, Copilot, or raw API calls without adaptation. You can extend Claude Code, but you don't really own the harness — the core prompts, behaviors, and tool wiring are closed-source and opinionated. If Anthropic changes the hook API, Meridian's hook layer breaks.

**2. It can't fix bad specs.** The gate model enforces that a `CONTRACT.md` exists and has the required fields. It can't tell you if the contract is *wise*. Garbage in, gated garbage out. The human's job of specifying what to build clearly is not something the harness can do for you.

**3. Real-time systems and hardware interfaces are out of scope.** Meridian is a file/code/document harness. It can't gate-enforce a firmware flash, a physical sensor calibration, or a live trading execution. For anything with physical-world or real-time consequences, the harness is advisory, not enforcement.

**4. The token cost is real.** The Gate Evaluator is a full Claude API call at every gate transition. A project with 5 gates and 3 iterations per gate is 15 evaluator calls on top of the coding agent calls. On a large project this adds up. The telemetry layer will surface this honestly — but it's not free.

**5. Context anxiety isn't fully solved on smaller models.** Claude Sonnet 4.5 exhibited context anxiety strongly enough that compaction alone wasn't sufficient — context resets became essential. But the same harness on Opus 4.5 found the behavior was gone. Meridian's Initializer/Coding Agent split mitigates this, but on Haiku or smaller models, long sessions may still degrade.

**6. It's a discipline enforcer, not an intelligence amplifier.** Meridian makes a capable model more reliable. It does not make a weak prompt produce a strong result. The quality of your SPEC.md, CONTRACT.md, and FEATURES.json determines the ceiling.

## How to Actually Use It: A Project Walkthrough

Let's use AeroIntel as the concrete example — it's real, it's yours, and it has natural gates.

---

**Step 0: Install**

```bash
git clone https://github.com/PCSchmidt/meridian
cd your-aerointel-repo
bash ../meridian/install.sh --recipe ml-pipeline
```

Install copies the hook files into `.claude/hooks/`, the skills into `.claude/skills/`, the subagent definitions into `.claude/agents/`, and writes the skeleton `.meridian/gates.yaml` for the ml-pipeline recipe. Runs `meridian-doctor.sh` to verify installation.

---

**Step 1: Define Your Contract (You, Not the Agent)**

Before you open Claude Code, you write `CONTRACT.md` yourself. This is the human's irreplaceable job:

```markdown
# AeroIntel Data Contract

## Data source: OpenSky Network REST API
## Schema: {icao24, callsign, origin_country, time_position, 
            longitude, latitude, baro_altitude, velocity, heading}
## Target: Anomaly flag (boolean) on each flight state vector
## Evaluation metric: Precision ≥ 0.85, Recall ≥ 0.70 at 5% contamination rate
## Baseline: IsolationForest with default parameters, no Kalman filtering
## Deployment target: Railway (backend), Vercel (frontend)
## Stack: FastAPI, Next.js 14, MapLibre GL, WebSocket
```

This is 10 minutes of your thinking. It's the most important 10 minutes in the project.

---

**Step 2: Initialize**

```bash
claude  # opens Claude Code in aerointel/
/init
```

The Initializer Agent reads `CONTRACT.md` and:
- Expands it into `FEATURES.json` with every feature marked `passing: false` — Kalman filter implementation, DBSCAN clustering, IsolationForest, WebSocket connection, MapLibre integration, each as a separate atomic feature
- Writes `PROGRESS.md` with session 1 started
- Writes `SPEC.md` expanding the contract into detailed technical specification
- Sets up `DECISIONS.md` template for architectural choices
- Creates `ASSUMPTIONS.md` with Meridian's pre-populated entries plus three AeroIntel-specific ones (why Kalman over raw EKF, why DBSCAN over k-means, why IsolationForest over One-Class SVM)
- Makes initial git commit: "chore: meridian init, features scaffolded (0/22 passing)"
- Runs `meridian-doctor.sh`, confirms clean

You read `SPEC.md` and `FEATURES.json`. If anything is wrong, you edit them now — before a single line of implementation code exists.

---

**Step 3: Clear Gate 0 (Confirmed)**

```bash
/review CONTRACT.md SPEC.md
# spec-reviewer subagent runs, reports gaps
# You address them
# You type the approval token:
CONFIRMED
```

The `validate-contract.sh` hook (optionally: `pyshacl` against `contract.shacl.ttl`) passes. Gate 0 clears. `CONFIRMED_GATE.json` is written. The hook writes a telemetry event. The reflexion system records "Gate 0 confirmed — estimated 0 revision cycles, actual 1."

---

**Step 4: Build, One Feature at a Time**

```bash
/continue
```

The Coding Agent reads `PROGRESS.md` and `FEATURES.json`. Picks `f001: Kalman filter state estimation`. Implements it. Writes tests. Runs tests (automated gate — tests must pass before `/continue` can write to `FEATURES.json`). Updates `f001: passing: true`. Commits: `"feat(f001): Kalman filter — 5-state vector, process/measurement noise tunable"`. Updates `PROGRESS.md`.

You run `/continue` again. It picks `f002: DBSCAN clustering`. Implements. Tests. Commits.

You are not involved in implementation decisions. You are the spec author and the gate approver. The agent is the builder.

---

**Step 5: Gate Evaluator Fires at Each Gate**

When the agent attempts to clear a gate, `run-evaluator.sh` fires before the approval token is accepted:

```json
{
  "gate": "pipeline_validated",
  "scores": {
    "completeness": 8,
    "quality": 7,
    "consistency": 9,
    "spec_adherence": 8
  },
  "overall": 8.0,
  "issues": [
    {
      "artifact": "kalman_filter.py",
      "severity": "medium",
      "description": "Process noise matrix Q is hardcoded — CONTRACT.md specifies it must be tunable via config"
    }
  ],
  "recommendation": "PASS_WITH_WARNINGS"
}
```

The hook writes this to `ERRORS.md`. The agent sees it, fixes the hardcoded Q matrix, re-runs. Gate clears on the second attempt. The reflexion system records the pattern: "Hardcoded parameters in ML modules — add to CLAUDE.md as an explicit never-do."

---

**Step 6: `/health report` Mid-Project**

After 3 sessions:

```
Gate pass rates:     confirmed: 1/1 (100%), pipeline_validated: 1/2 (50%)
Avg phase duration:  confirmed: 0.4h est, 0.4h actual (1.00x)
                     pipeline_validated: 4h est, 6.5h actual (0.62x)
Most common blocks:  hardcoded_parameters (2x), missing_test_coverage (1x)
Token consumption:   session 1: ~18k, session 2: ~24k, session 3: ~21k
Evaluator avg score: 7.8/10
Est. project cost:   $4.20 so far (3 sessions × avg 21k tokens)
```

You now know: your pipeline phase estimates are off by 38%. The evaluator is catching hardcoded parameters repeatedly. You add one line to `CLAUDE.md`: "Never hardcode numerical parameters in ML modules — all must be configurable via constructor arguments or config file." And you add it to `ASSUMPTIONS.md` with the failure mode citation.

---

**Step 7: `MODEL_EVAL` Gate — The Human Decision**

The automated benchmark runs: Precision 0.87, Recall 0.73 at 5% contamination. Beats the baseline (IsolationForest default: Precision 0.79, Recall 0.61). The Gate Evaluator scores the eval report: 9.0/10, `PASS`.

But you review the numbers yourself. You notice recall on cargo aircraft is only 0.61 — below the contract threshold for that subclass. You reject the gate manually. You add `f023: improve cargo aircraft recall` to `FEATURES.json`. The agent implements it. Recall rises to 0.74. Gate clears.

This is the harness working as designed: the automated gate catches obvious failures, the human gate catches domain-specific judgment calls.

---

**Step 8: Deploy**

```bash
/deploy
```

The deploy skill checks `DEPLOY_APPROVED` artifact exists (the lock set at Gate 1). Generates Railway and Vercel deployment configs. Runs `git tag v0.1.0`. Pushes. AeroIntel is live.

Total: 8 sessions across 3 days. `PROGRESS.md` has a complete session-by-session record. `FEATURES.json` shows 22/22 passing. `ASSUMPTIONS.md` has 8 entries — 5 Meridian defaults plus 3 AeroIntel-specific ones you added. The benchmark shows 10% precision improvement and 20% recall improvement over baseline. That number goes in the portfolio README.
