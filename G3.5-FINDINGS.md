## Finding 1: The "90% done" illusion is real and measurable

| Metric | Before correction | After drift sensor fix |
|--------|-------------------|------------------------|
| Happy-path complete | 92% (12/13) | **100% (13/13)** |
| Full-lifecycle complete | **0% (0/13)** | **0% (0/13)** |

AeroIntel is a live production project deployed on Fly.io + Vercel, with Phase 5 features complete
and evidence captured. It *feels* done. The happy-path number confirms it: 100% of features work
under normal conditions. But full-lifecycle is 0% — zero features have been hardened through
edge cases, failure modes, and demo-stability testing.

**The gap is real, not cosmetic.** Edge cases (WebSocket reconnection, aircraft dropping from
state, IsolationForest pre-warm period, Claude API timeouts) exist and are undocumented.
Hardening (30-minute demo stability, load behavior) was verified empirically but not encoded in
the tracker. This is exactly the state that misleads engineers into shipping.

## Finding 2: Drift sensor was accurate, advisory signal was correct

Drift verdict: **alignment_score 7 / aligned** — no scope creep, no contradictions.

One medium `feature_lag` was flagged: Portfolio Evidence Package marked 0% while
CASE_STUDY.md and 4 anomaly JSONs already existed. The sensor caught a real staleness —
the FEATURES.json hadn't been updated to reflect work already completed. Correcting this
raised happy-path from 92% to 100%.

**Drift sensor verdict:** Ready to promote to warn-blocking (advisory warning on drifted
recommendation). One real project validation with clean aligned verdict and one caught staleness
is sufficient signal. No false positives on the aligned fixture.

## Finding 3: Installer works; two rough edges found

### Rough edge 1: gates.yaml path mismatch

First install attempt placed `gates.yaml` at project root (`aerointel/gates.yaml`).
`gate-engine.sh` expects it at `.meridian/gates.yaml`. Fixed in `install.sh` before
the aerointel install completed cleanly.

**Impact:** Low. Fixed before the actual install. No corrective action needed in user projects.

### Rough edge 2: SPEC.md boilerplate heading

`features-init.sh` picked up `## Feature List` as a feature (14 features instead of 13).
Required removing the `##` heading from SPEC.md. The filter in `features-init.sh` doesn't
include "Feature List" in its boilerplate exclusion list.

**Impact:** Low. Fixable in `features-init.sh` by adding "Feature List" to the exclusion list,
or by SPEC.md authoring discipline (don't use `##` for non-feature section headers).

**Preferred fix:** Add `"Feature List"` and `"Features"` to the boilerplate filter in
`features-init.sh`. Users shouldn't need to know about this constraint when writing their spec.

### Rough edge 3: No CLAUDE.md for the target project

The installer does not create a `CLAUDE.md` in the target project. Engineers starting a new
session in AeroIntel won't get the Meridian context (gate model, where things live, etc.)
unless they write one manually.

**Impact:** Medium. The hooks and skills are installed and will run, but an agent starting a
session in the project won't know about the gate model or how to use Meridian's tools.

**Preferred fix:** Ship a `templates/CLAUDE.md` in Meridian and have `install.sh` copy it
to the target project (only if not already present). The template would explain the gate model
and point to `.meridian/` and `.claude/`.

## Finding 4: Lifecycle tracker provides better signal than drift sensor on depth

| Tool | AeroIntel finding | Signal quality |
|------|-------------------|----------------|
| Drift sensor | 7/10 aligned, 1 medium feature_lag | Correctly showed no scope creep |
| Lifecycle tracker | 100% happy-path / 0% full-lifecycle | **Stark — immediately actionable** |

The lifecycle tracker's two-number summary (happy-path vs full-lifecycle) is the more
impactful tool for an engineer looking at this project. The 100/0 split creates an immediate
conversation: "we're done but we're not done." The drift sensor is the right tool for a
different failure mode (scope creep during active development).
